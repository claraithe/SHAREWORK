# Contributing to SHAREWORK

Thank you for your interest in contributing to SHAREWORK! This document provides guidelines for contributing to the project.

## How to Contribute

### 1. Share Your Work Item Examples

Have a great example of collaborating with ChatGPT? Share it!

1. Create a new file in the `examples/` directory
2. Use the format: `example_work_item_[number]_[topic].md`
3. Follow the work item template structure
4. Include ChatGPT collaboration notes showing the interaction
5. Submit a pull request

### 2. Improve Documentation

Help make the documentation better:

- Fix typos or unclear explanations
- Add missing information
- Improve examples
- Suggest better practices
- Add new use cases

### 3. Suggest New Templates

If you have ideas for additional templates:

- Share your template idea
- Explain the use case
- Provide an example
- Submit via pull request or issue

### 4. Share Best Practices

Discovered a great way to collaborate with ChatGPT?

- Document your approach
- Add it to the COLLABORATION_GUIDE.md
- Or create a new guide if it's substantial
- Share your success stories

## Contribution Guidelines

### Quality Standards

- **Clear and Concise**: Write clearly and avoid jargon
- **Well-Structured**: Follow existing document structures
- **Practical**: Include real-world examples
- **Tested**: Ensure examples work as described

### Documentation Style

- Use Markdown formatting
- Include code blocks with language specification
- Use headers for organization
- Add emoji sparingly for visual interest
- Keep line length reasonable (80-100 characters when possible)

### Example Contributions

**Good Example:**
```markdown
## Pattern: Debugging API Issues

1. Share the API endpoint code
2. Include the error response
3. Show request/response examples
4. Ask ChatGPT to identify the issue
5. Implement the suggested fix
6. Test and verify

### Real Example:
[Complete, working example with actual code]
```

**Needs Improvement:**
```markdown
Just ask ChatGPT to fix bugs.
```

## Pull Request Process

1. **Fork** the repository
2. **Create** a branch for your changes (`git checkout -b improve-documentation`)
3. **Make** your changes
4. **Test** that all links and examples work
5. **Commit** with clear messages (`git commit -m "Add debugging pattern to collaboration guide"`)
6. **Push** to your fork (`git push origin improve-documentation`)
7. **Submit** a pull request

### PR Description Template

```markdown
## Description
[What does this PR do?]

## Type of Change
- [ ] Documentation improvement
- [ ] New example
- [ ] New template
- [ ] Bug fix
- [ ] Other: [describe]

## Testing
[How did you verify your changes work?]

## Additional Notes
[Any other context or screenshots]
```

## Code of Conduct

### Our Standards

- Be respectful and inclusive
- Welcome newcomers
- Accept constructive criticism
- Focus on what's best for the community
- Show empathy towards others

### What We Don't Accept

- Harassment or discrimination
- Trolling or insulting comments
- Personal or political attacks
- Publishing others' private information
- Other unethical or unprofessional conduct

## Getting Help

Need help with your contribution?

- **Questions**: Open an issue with the "question" label
- **Discussion**: Use GitHub Discussions
- **Problems**: Open an issue with details

## Recognition

Contributors will be:
- Listed in the project (if desired)
- Credited in pull requests
- Thanked in release notes
